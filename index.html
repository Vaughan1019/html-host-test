    Foundations

    The ground floor. If APIs are new to your team, start here. Otherwise, mark these off and move on.

    What you need to know

    What is an API?

    An API is how two software systems talk to each other directly — no human clicking buttons. Instead of typing a referral into a web form, your system sends structured data to SIGHTLINE and gets a structured answer back.

    Think of it like ordering at a counter: you hand over a form (the request), the kitchen prepares it (SIGHTLINE processes), and you get a receipt back (the response).

    HTTPS — Encrypted in transit

    Every call to SIGHTLINE travels over HTTPS, which encrypts data so it can't be read in transit. If you can access a secure website, you already understand the concept.

    JSON — The format

    JSON is a lightweight text format for structured data. It looks like this:

    {
  "firstName": "Jane",
  "lastName": "Doe",
  "age": 42,
  "isActive": true,
  "phones": ["555-0100", "555-0142"]
}

    Curly braces for objects, quotes for text, brackets for lists. A SIGHTLINE case is the same shape, just more fields.

    The flow

    1. Your system authenticates and gets a temporary token
    2. Your system sends a POST request with JSON + token
    3. SIGHTLINE validates and replies with a JSON result
    Ready for Step 2 when…You understand: your system sends JSON over HTTPS, gets JSON back.

    Checklist

    • I understand what an API is
    • I know HTTPS = encrypted communication
    • I can read a basic JSON object
    • I understand request → response flow

    Connect & Authenticate

    Get your credentials, request a token, and prove connectivity. This is where integration begins.

    Concept

    OAuth 2.0 Client Credentials

    Machine-to-machine authentication. You have a Client ID (app username) and Client Secret (app password). Trade them for a short-lived bearer token. Key facts:

    • No refresh tokens — when it expires, request a new one
    • Tokens are short-lived — cache in memory and use the expires_in value from the token response to refresh before expiry (confirm the exact lifetime with Ethos).

    Prerequisites

    • Developer assigned
    • Client ID, Client Secret, and Scope received from Ethos
    • HTTP client installed (Postman, curl, etc.)
    • Network allows HTTPS to idp.staging.atlas.ethosrisk.com

    Your scope

    atlas/global/allow
atlas/<client-name>/parentcompanyid:xxxxx
atlas/<client-name>/companydivisionid:xxxx
atlas/<client-name>/clientid:xxxxx
atlas/<client-name>/userid:xxxxx

    Request a token

    curl -X POST https://idp.staging.atlas.ethosrisk.com/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET" \
  -d "scope=atlas/global/allow atlas/yourco/parentcompanyid:12345"

{

    ["client_secret"] = Environment.GetEnvironmentVariable("SL_SECRET"),
    ["scope"] = "atlas/global/allow atlas/yourco/parentcompanyid:12345"
});
var resp = await httpClient.PostAsync(
    "https://idp.staging.atlas.ethosrisk.com/oauth2/token", content);
var token = JsonDocument.Parse(await resp.Content.ReadAsStringAsync())
    .RootElement.GetProperty("access_token").GetString();

    Response

    { "access_token": "eyJhbGciOi...", "token_type": "Bearer", "expires_in": 3600 }

    Use it

    Authorization: Bearer eyJhbGciOi...
    Security JWTs are signed, not encrypted — treat as credentials. All traffic MUST be HTTPS. Store Client Secret in a secrets manager. No refresh tokens issued; re-auth on expiry.

    Harden your auth

    • Validate the token before trusting it — check the iss (issuer), aud (audience), and exp/nbf (expiry / not-before) claims.
    • No refresh token is issued in the client-credentials flow. When the token expires, request a new one — don't attempt a refresh grant.
    • Rotate the Client Secret on a regular cadence and immediately if it may be exposed. Confirm the rotation cadence and whether a dual-secret (overlap) window is supported with Ethos.
    • If a secret or token is ever seen over plain HTTP, treat it as compromised and rotate it.

    Checklist

    • Prerequisites complete — credentials in hand
    • Successfully obtained a bearer token
    • Understand token lifecycle (obtain → cache → refresh)
    • Secret stored securely (not in source control)

    Submit Your First Case

    Build a CaseAssignment JSON, POST it to SIGHTLINE, and read the result. This is the core integration.

    Concept

    What's a "staged case"?

    Submissions to /api/v3/stagedcases are not final — they require manual review and approval by Ethos operations before being promoted to an active case. You receive a ConfirmationNumber immediately — use it to follow up.

    Don't confuse the terms: a staged case is a submission awaiting review; the staging environment is the test/sandbox platform. They're unrelated.

    Getting the outcome back Today, case results and updates are returned out-of-band — there is no GET, status-check, or webhook endpoint. Keep the ConfirmationNumber and follow up with Ethos staff. Confirm the outcome-delivery mechanism with Ethos during onboarding.

    EnvironmentURL

    Dates & times Date-only fields (DueDate, DateOfLoss, DateOfBirth) use ISO 8601 yyyy-MM-dd. For timed fields, DueTime is interpreted as Eastern Time, and DueTimeUTC takes precedence when both are supplied. Strings over 100 characters (non-comment fields) may be truncated.

    Success

    { "StatusCode": 0, "ConfirmationNumber": "ETH-2026-A4C2F1", "Messages": [] }

    { "StatusCode": 101, "ConfirmationNumber": null,

    Unauthorized

    { "StatusCode": 201, "ConfirmationNumber": null,
  "Messages": ["Token expired or invalid."] }
    NoteStatusCode 201 means "Unauthorized" here — it's an application code, not HTTP 201 Created. Always check the JSON body.

    Attachments — two methods

    MethodStatusWhen to use
    Direct payload (file embedded in JSON)LegacyExisting integrations only
    Pre-signed URL uploadRecommended for new clientsBetter performance, no JSON body size limits

    Pre-Signed URL upload

    New in v1.4

    Recommended for new clients

    Upload files directly to a storage endpoint instead of embedding them in the payload — better performance and reliability, and it keeps large files out of the JSON body.

    Opt-in required Pre-signed URL upload is enabled per client. To turn it on: contact your Ethos Account Manager or API support team → request enablement → receive updated guidance, including the endpoint/flow details and the required request structure.
    1. Client requests the ability to upload an attachment
    2. System generates a pre-signed URL for the upload
    3. Client uploads the file directly to storage using that URL
    4. Client submits the Stage Case request including a reference to the uploaded file (URL / key / identifier)
    5. SIGHTLINE processes the case and associates the attachment

    Compliance & data protection

    Production cases carry PHI / PII — names, dates of birth, and medical details. Handle them under your Statement of Work and any associated data-protection agreement (BAA):

    • Encrypt in transit — every call over HTTPS (TLS). Confirm the minimum TLS version, at-rest encryption, and key-management with Ethos.
    • Apply the minimum-necessary principle — send only the fields a case needs.
    • Restrict access on a need-to-know basis.
    • Never route production data through the staging environment.
    Confirm with EthosRequired-vs-optional PHI fields, what is returned vs withheld in responses, audit-logging, and the incident / breach-notification path are confirmed during onboarding.

    Pre-go-live checklist

    • Production credentials + endpoint confirmed
    • Claim/service lists validated in production
    • All status codes handled (0, 101, 111, 201, 999)
    • Secret in secrets manager
    • Token lifecycle managed
    • Attachment strategy implemented
    • Security review passed
    • First production case submitted

    SSO & Just-in-Time Provisioning

    Optional. Let your users access SIGHTLINE through your identity provider with auto-created accounts.

    What's supported today SIGHTLINE supports OIDC / JIT via Microsoft Entra. If your identity provider uses a different method, contact your Ethos Account Manager — we'll review your setup and handle it separately.
    Concept

    SSO + JIT

    SSO: Users log in once through your IdP (Azure AD, Okta) and are recognized by SIGHTLINE.

    JIT: First-time SSO users get auto-created with correct permissions. No tickets.

    How it works

    1. User signs in via your IdP
    2. Token includes a custom claim (user type)
    3. SIGHTLINE maps claim → client/division
    4. New user? Auto-create, attach, sign in.
    5. Disable in your IAM → access revoked immediately

    1. Confirm the custom claim / attribute name and the allowed values with Ethos.
    2. In staging, sign in via your IdP → confirm JIT auto-creates the account and lands on the correct division / service list.
    3. Test each user-type value → verify the mapped capability (Restricted / Standard / Administrator).
